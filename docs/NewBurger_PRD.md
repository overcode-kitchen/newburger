## 2. 페이지 구조

**콘텐츠 역할:** **메뉴(사실 정보)**는 운영이 일관되게 입력·관리하고, **후기·별점**은 방문자 UGC로 쌓인다. (상세 원칙은 **§10**.)

```
뉴버거
├── 홈 (신메뉴 피드)          → 메인 진입점
├── 메뉴 상세 (/menu/[id])    → 상세 정보 + 후기
└── 어드민 (/admin)           → 내부 전용 (비공개) — 운영 빈도에 따라 도입 (§10)
```

## 8. 데이터 구조

### Menu
```
- id
- brand (mcdonald / burgerking / lotteria / moms)
- name (메뉴명)
- price_single (단품 가격)
- price_set (세트 가격 — null이면 미기재·미판매 등)
- image_url
- release_date (출시일)
- end_date (판매 종료일 — null이면 상시)
- is_limited (한정 여부)
- calories (칼로리)
- official_link (공식 링크)
- description (설명)
```

### Review
```
- id
- menu_id
- rating (1–5)
- comment (텍스트 후기 — 선택)
- taste_vote (맛있다 / 애매하다 / 맛없다 — 선택)
- value_vote (좋다 / 보통 / 아쉽다 — 선택)
- repurchase_vote (있다 / 고민 / 없다 — 선택)
- ip_hash (중복 방지용)
- created_at
```

### Supabase·환경 (오해 방지)

- **연동·스키마**가 반영되어 있으면 “백엔드 미완(🟡)”이 아니라 **연동 완료(✅)**로 본다.
- **로컬:** `.env.local`에 `NEXT_PUBLIC_SUPABASE_URL`·`NEXT_PUBLIC_SUPABASE_ANON_KEY` 설정 **해결됨** (2026-04 기준).
- **배포:** 호스팅(Vercel 등)에도 동일 키를 넣어야 하며, 미설정 시 빈 목록·안내는 **설정 누락**에 따른 동작이다.

## 10. 운영·신뢰 원칙 (메뉴 vs 후기·어드민)

### 역할 분리

| 구분 | 성격 | 누가 채우나 | 비고 |
|------|------|-------------|------|
| 메뉴 | **사실 정보** (이름·가격·이미지·기간 등) | 운영자 — 어드민·DB 직접 입력·시드 SQL 등 | 일관된 카탈로그가 전제 |
| 후기·별점 | **의견·평가** | **여러 방문자(UGC)** | 서비스 신뢰의 중심은 “다수의 실제 후기” |

운영자만 후기를 대량으로 쓰면 **브랜드 블로그**에 가깝고, 사용자 입장에서 **신뢰는 크게 오르지 않는다.** 제품 목표와 직결되는 것은 **후기가 여러 사람 것으로 보이게 하는 것**이며, 그다음이 유입·공유·반복 방문으로 **표본 수 N을 쌓는 것**이다.

### 어드민 (`/admin`) 도입 타이밍

- **신메뉴 등록 빈도가 낮다** (월 소수 건): Supabase 테이블 직접 입력·`seed`/SQL로도 운영 가능. 이 경우 **어드민은 급하지 않다.**
- **등록이 잦다** (매주 여러 건·여러 명이 입력): 실수·일관성 문제가 커지므로 **간단한 등록 UI + 접근 통제(인증)** 또는 `/admin`을 **일정을 잡고** 만드는 편이 낫다.

**요약:** “지금 당장 어드민”은 **운영 빈도·팀 규모**에 맞추고, **다수 UGC 후기 + 이를 보여주는 UX** 방향은 유지한다.

### 평가·신뢰 기능 로드맵 (단계)

1. **MVP~초기:** 익명 별점·텍스트 후기·후기 수·최신순 노출. 평균 별점은 **표본이 적을 때 왜곡**될 수 있음을 인지한다.
2. **후기가 쌓인 뒤:** 표본 수 강조, **최소 N건 이상일 때만** 대표 별점을 크게, **분포**(5·4·3…), **최신성 가중** 등은 **순차 도입**이 현실적이다.
3. **구조화 후기 신호 도입:** 별점·텍스트 외에 `맛 만족도/가성비/재구매 의사`를 **선택 입력**으로 추가하고, 상세에서 n명 기준 비율로 요약 노출한다.
4. **규칙 집계 보조:** 입력 신호가 비어 있는 텍스트 후기는 키워드 사전 기반 규칙 집계를 보조로 사용한다. 단, UI에서 **직접 선택 기반**과 **텍스트 추론 기반**을 구분 표기한다.
5. **남용·중복:** 현재 `ip_hash`는 로그인 없는 단계에서의 **완화책**이며, 완벽한 1인 1표는 아니다. 신뢰·거버넌스를 올리려면 이후 **로그인**, **신고/숨김**, **비정상 패턴 대응** 등을 검토한다.

**한 줄:** 신뢰의 심장은 **다수의 실제 후기**이고, **정교한 점수 알고리즘은 그다음 레버**다.

## 11. PRD 문서 관리 (어떻게 갱신할까)

이 파일은 **(A) 제품 스펙·기획 확정**과 **(B) 특정 시점 구현 스냅샷**을 함께 담는다. 섞이면 혼란이 나므로 역할을 구분한다.

| 구분 | 무엇을 쓰나 | 갱신 시점 |
|------|-------------|-----------|
| **본문 §3~9, §10** | 무엇을 만들지·무엇을 약속하는지(체크는 **기획·스펙 합의** 기준) | 기획 변경·합의 시 |
| **상단 「구현 진행 현황」표** | 그 시점 **코드/배포 기준**으로 무엇이 됐는지 | 릴리스 전·스프린트 끝·큰 기능 머지 후 등 |
| **§10** | 운영·신뢰 **원칙**(PO 관점) — 스펙과 동급으로 유지 | 방향 전환 시 |

**권장 습관**

- 기획만 바뀌면: **본문·§10**만 고치고, 구현 표는 나중에 한 번에 맞춰도 된다.
- “해결됨”을 본문에 남길지 말지: 본문 체크는 **「기획서상 요구 충족」**으로 두고, **실제 배포 여부**는 상단 표 또는 이슈/릴리스 노트에 맡기면 **스펙 문서가 구현 메모로 오염되지 않는다.** (작은 팀이면 상단 표만 주기적으로 갱신해도 충분하다.)
- 버전: 제목의 **v1.x**와 하단 **변경 이력**으로 큰 수정을 남긴다.
